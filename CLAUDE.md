# Challenge Hub — Agent Handover

Read this before writing any code. It is the ground truth for the current state of the project.

## What this project is

A Next.js 14 web platform that lets small friend groups create and compete in fitness challenges, with automatic activity sync from Strava. Initial audience: ~15 people, soft ceiling of a few hundred. Simplicity and low operational cost are explicit priorities.

| Concern | Choice |
|---------|--------|
| Framework | Next.js 14, App Router, TypeScript strict |
| Styling | Tailwind CSS + shadcn/ui |
| Auth | Firebase Authentication (email/password + Google) |
| Database | Firestore (fresh project — no migration needed) |
| Storage | Firebase Storage (profile photos only) |
| Hosting | Netlify via `@netlify/plugin-nextjs` |
| CI/CD | GitHub Actions |
| Testing | Vitest + React Testing Library |
| Activity source | Strava only (Garmin is Phase 2) |
| State | SWR for server state, React Context for auth only |

### Done in session 11 (post-Strava polish + PWA + security review)
- [x] **Security review of the Strava/account-deletion/PWA surface added since session 8** — ran a structured review (candidate findings → independent verification pass per finding) rather than a single pass, specifically to catch false positives. Found and fixed two real, confirmed issues on the **already-live, already-registered** webhook — treated as urgent, fixed same-session rather than just reported:
  - **Unauthenticated webhook let forged events delete other users' data.** `POST /api/strava/webhook` had zero verification on the request body (matches Strava's real protocol — no per-event signature — but the app added no compensating control). `syncStravaActivity`'s create/update path already re-fetches the canonical activity from Strava before writing anything, but `removeStravaActivity` (delete events) and the athlete-deauthorization path did not — they trusted the webhook body outright. A forged `{aspect_type:"delete", owner_id, object_id}` for any known `stravaActivityId` (which the app's own activity feed exposes to fellow challenge members via a link to strava.com) could silently strip a real, still-existing activity from someone else's leaderboard totals; a forged athlete-deauthorization event (needs only a public Strava athlete id) could disconnect anyone's integration. **Fixed**: `removeStravaActivity` now re-fetches the activity from Strava first and only proceeds if it's genuinely gone (404 → `fetchStravaActivity` now returns `null` instead of throwing, so "confirmed gone" is distinguishable from a transient error); `syncStravaActivity` does the same self-healing cleanup if a create/update event arrives for an activity that's since been deleted. Also added an optional `STRAVA_SUBSCRIPTION_ID` env var check (defense-in-depth for the deauth case specifically, which can't be re-verified the same way since a deauthorized token can't be used to ask Strava anything) — **Andreas needs to set this** to the numeric id Strava returned when the webhook subscription was registered (`GET https://www.strava.com/api/v3/push_subscriptions` with client_id/secret shows it if he didn't save it), in both `.env.local` and Netlify. Works without it, just without that one extra layer.
  - **Race condition let concurrent syncs double-count a single activity.** `reconcileGoalActivity`/`reconcileZoneActivity` did a plain read-then-batch-write (query for an existing entry, then a separate `batch.commit()`) — not atomic, so two near-simultaneous `syncStravaActivity` calls for the same activity could both see "no existing entry" and both apply a positive score delta. Confirmed practically exploitable (not just theoretical): the multi-await path before reconciling (token refresh, Strava API calls) creates a real network-latency-sized window, and — combined with the unauthenticated webhook above — anyone could trigger it deliberately with a couple of parallel `curl` requests against their own real activity id, inflating their own leaderboard score. **Fixed**: both functions now run inside a Firestore transaction (`db.runTransaction`), which Firestore automatically retries if the read data changes before commit, closing the race properly instead of just narrowing it.
  - A third candidate finding (OAuth `state` signing fails open to a guessable key if `STRAVA_CLIENT_SECRET` is ever unset) was investigated and ruled a **false positive** on verification: the same env var is independently required by Strava's own token-exchange endpoint in the same flow, so an unset secret breaks the flow visibly long before this specific gap could be exploited. Correctly filtered out rather than reported as a fix.
